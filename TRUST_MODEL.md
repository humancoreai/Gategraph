# GateGraph Trust Model

Version: v0.17.2_CANDIDATE  
Base: v0.17.2_CANDIDATE  
Scope: caller trust boundary and release integrity clarification

## 1. Purpose

This document defines what GateGraph trusts, what it validates, and what it does not claim to verify.

GateGraph is a deterministic governance and enforcement layer. It evaluates explicit task metadata against configured rules, budgets, and guards. It does not semantically inspect task content to determine whether caller-provided metadata is true.

## 2. Trust Boundaries

GateGraph has one primary external trust boundary: the caller request boundary.

A caller submits a task request containing capability intent and security-relevant metadata. GateGraph validates required structure, records the supplied values, and evaluates them deterministically. GateGraph does not infer missing security meaning from task text or payload content.

## 3. Caller-Controlled Fields

The following fields are caller-controlled security metadata:

| Field | Required at adapter boundary | Meaning | Trust status |
|---|---:|---|---|
| `input_source` | yes | Caller-declared source category of the input | trusted caller metadata |
| `data_sensitivity` | yes | Caller-declared sensitivity category | trusted caller metadata |
| `secrets_involved` | yes | Caller-declared boolean indicating secret involvement | trusted caller metadata |
| `requested_capabilities` | yes | Caller-requested capabilities | caller request, structurally validated |
| `projected_cost_units` | optional | Caller-provided cost estimate used for budget reservation | caller input, structurally converted |
| `actor_id` | optional | Caller or adapter actor identity | caller/config boundary value |

GateGraph may reject missing or malformed required boundary fields. It does not determine whether the declared values are semantically truthful.

## 4. Internally Generated Fields

The following values are generated by GateGraph:

| Field | Origin |
|---|---|
| `correlation_id` | generated if absent |
| `event_id` | generated by governance evaluation |
| `decision_id` | generated by governance evaluation |
| `risk_level` | produced by deterministic risk rules |
| `risk_reason` | produced by deterministic risk rules |
| `selected_rule_id` | produced by deterministic rule evaluation |
| `matched_rule_ids` | produced by deterministic rule evaluation |
| `capability_token` | generated only after allow/warn/review-capability resolution |
| `budget_scope_id` | generated or selected by budget ledger |
| `budget_reservation_id` | generated by budget reservation |
| audit events | append-only event logger |

## 5. Security-Critical Metadata

The fields `input_source`, `data_sensitivity`, and `secrets_involved` are security-critical because they influence risk classification and rule matching.

In v0.16.0 the public adapter boundary requires these fields explicitly. Missing fields fail closed. Malformed boundary fields fail closed. Unknown enum values are not silently treated as safe by the risk engine.

Recognized `input_source` values are `local`, `external`, and `untrusted`. `trusted` is reserved for trust-level terminology and `TrustedEntryContext`; it is not a valid `input_source` value in the current release.

## 6. Boundary Validation

GateGraph validates:

- request body is a JSON object
- `requested_capabilities` is a list
- `input_source` is present and a non-empty string
- `data_sensitivity` is present and a non-empty string
- `secrets_involved` is present and a boolean

GateGraph does not automatically correct, enrich, downgrade, upgrade, or infer caller metadata.

## 7. Threats Not Mitigated

GateGraph does not mitigate these threats by itself:

- a trusted caller intentionally lying about `data_sensitivity`
- a trusted caller intentionally hiding `secrets_involved`
- a compromised upstream classifier providing false metadata
- semantic misclassification of task content before GateGraph receives the request
- prompt-injection inside task content unless represented through explicit metadata/rules
- distributed consensus failures
- multi-node replication conflicts
- malicious modification of local source files outside release verification

These threats must be handled by upstream controls, deployment policy, review process, or additional systems outside the current GateGraph scope.

## 8. Determinism Guarantees

For the same database state, rules, budgets, configuration, and request fields, GateGraph decision logic remains deterministic.

GateGraph does not use ML inference, semantic classifiers, adaptive policy updates, or autonomous policy repair in this boundary.

## 9. Audit Visibility Model

Caller-supplied boundary fields are recorded in the governance event input data. Manipulated caller fields are therefore visible as submitted values, not silently corrected values.

Audit visibility is not semantic truth verification. It proves what GateGraph received and evaluated, not whether the caller description was truthful.

## 10. Explicit Non-Goals

GateGraph does not claim:

- autonomous governance
- intelligent risk prediction
- automatic secret detection
- automatic sensitivity classification
- adaptive security policy generation
- caller truthfulness verification
- distributed or consensus-based trust
